## What it will do

Today, Orifold edits a page's **text** in place. Object editing extends that same "edit the real thing, not a copy floating over it" philosophy to a page's **graphics**:

- **Select** a line, arrow, rectangle, ellipse, image, logo, table border, or stamp by clicking it.
- **Move, resize, and rotate** it, with handles — the way you'd expect in a design tool.
- **Delete** it cleanly — the object actually leaves the file, with no ghost left behind.
- **Duplicate, align, and layer** (bring forward / send backward) supported objects.
- **Restyle** vector shapes — stroke color, fill color, line width, dashes, opacity.

Not every graphic on a page is a separate, editable object (some are flattened into a scan, some are drawn as one fused operation). Orifold will tell you clearly what it can and can't edit, and why, rather than silently doing nothing.

## Where it is now

<Callout variant="tip" label="Shipped: the hard part">
	The riskiest question — *can Orifold move and delete real page objects, then save and reopen the file with no leftover artifacts, no lost text, and each object still identifiable?* — has been answered **yes**, proven on real PDF bytes and locked in as a permanent test (shipped in v0.8.7). A read-only detection pass that recognizes a page's lines, shapes, and images and classifies what's safely editable also shipped. Both run entirely on your Mac, like everything else in Orifold.
</Callout>

The remaining work is the on-canvas experience — selection handles, dragging, the contextual toolbar, undo/redo, and reliable save/reopen — layered onto that proven engine. It stays **local-first, free, and open-source**, like the rest of the app.

<Callout variant="note">
	Until this ships, you can already reposition and remove things you added yourself — [stamps, watermarks, and signatures](/Orifold/annotate/stamps/) — and [edit text in place](/Orifold/edit/edit-text/). Object editing is about the graphics that were *already baked into* the page when you opened it.
</Callout>
